# 2026-07-24 性能与安全排查

起因:用户反馈"上下滚动页面稍微有点卡顿",顺带做一轮性能 + 安全全面扫描。
本文记录**已修**的结论和**留待后续**的问题,后者按"值不值得做"排了优先级。

实测方法沿用 `playwright-core` + `_electron.launch({args:['.']})`(见 memory `cover-particle-silk`);
注意应用会恢复上次会话状态,若上次退出时停在迷你播放条,主窗口不渲染,要先点 `[aria-label="回到大播放器"]`。

## 一、已修

### 1. 封面图片全站原图直出(滚动卡顿主因)

实测探索页 25 张图里 20 张是原图直出,最夸张的一张 **3000×3000 显示成 36px(41 倍浪费)**,
整页解码后像素内存 **138 MB**。滚动时新进视口的卡片要现场解码这些大图,就是"滚起来一顿一顿"的来源。

修法:所有列表/卡片/头像封面统一走 `sizedImage(url, 显示尺寸×2)`(`src/lib/image-size.ts`,
网易图床 `?param=WxH`、QQ 图床改路径档位)。覆盖 17 处 `<img>` + canvas 侧取图。

实测:探索页 138.1 MB → **7.9 MB**,榜单页 11.9 MB → 4.2 MB,视觉无差别。

**约定**:新增封面 `<img>` 一律带 `sizedImage`,尺寸取 CSS 显示尺寸的 2 倍(Retina)。

### 2. 走 `/proxy/cover` 的取色/纹理请求一直被自己 403 掉(既有 bug)

Chromium 从 `file://` 页面发出的带 CORS 的请求(`crossOrigin='anonymous'` 的 `Image`),
Origin 头是 **`file://`** 而不是规范里的 `"null"` —— `isAllowedOrigin` 只认后者,
于是封面调色板取色、封面粒子纹理、3D 舞台取色的代理请求全部 403。
表现:霞光/标题渐变/粒子恒为默认紫色,**从来没跟过封面**。已在 `server/lib/security.ts` 放行 `file://` 并补测试。

修完后同一页面的标题色从默认紫变成随封面的暖橙,肉眼可见。

### 3. canvas 侧三处取图尺寸不一致 → 同一封面连打三次上游

改动 1 之后,取色(96)、3D 取色(128)、粒子纹理(384)变成三个不同 URL,切歌瞬间对同一张封面
连发 3 个代理请求,网易图床按频率 403 限流。统一到 `CANVAS_COVER_PX = 384` 一档,复用同一个 URL。

另外 `track-preload` 的封面预热此前套了 `api.coverImage()`(代理 URL),而播放条 DOM 用的是直连 URL,
**两个 URL 不同,预热等于白做**;已改成预热 DOM 实际使用的那个地址。

### 4. 榜单页预览"滚到哪加载到哪"

见同日另一处改动:`src/lib/toplist-cache.ts` 并发池 + 分组缓存,进页面即后台预取全部榜单 Top3。

### 5. 安全加固

| 项 | 结论 |
| --- | --- |
| 本地 API 来源校验 | 打包应用不再放行 `http://localhost:*`(本机其他网页/其 XSS 曾可读 `/api/local/tracks` 拿文件绝对路径、经 `/api/local/audio` 流走文件);dev 仍放行 |
| 写操作 CSRF | 打包应用下 POST 必须带 Origin,`<img>`/`<form>` 这类不带 Origin 的跨源加载再也触发不了扫盘/删歌单 |
| CSP | 渲染层 4 个入口 HTML 此前**完全没有 CSP**,已加(`script-src 'self' 'unsafe-inline'` 等);prod 实测 0 违规、0 报错 |
| 悬浮窗导航防护 | 桌面歌词/壁纸/迷你条三个窗口都挂着 preload 桥,但只有主窗口设了 `setWindowOpenHandler` + `will-navigate`,已补齐 |
| 权限请求 | Electron 默认"全部允许",已改为默认拒绝,只留 `fullscreen` 与 `clipboard-sanitized-write` |

已确认**没有**问题的点:`/api/audio`、`/proxy/cover`、`/api/cover`、播客音频代理都过了 `isSafeUpstreamUrl`;
`shell.openExternal` 有协议白名单;更新包下载要求 digest;全项目无 `eval` / `dangerouslySetInnerHTML`;
本地音乐按索引 id 取路径,不接受调用方传路径,无穿越。

## 二、留待后续

### P1 玻璃层 + 全屏 WebGL 的常驻 GPU 成本

探索页同屏 **12 个 `backdrop-filter` 元素**,身后是每帧变化的 LiquidEther 流体。
背景一动,所有玻璃层都要重新取样模糊,这是本项目 GPU 占用的大头(`AmbientBackground.module.css`
里已有一条相关教训:eco 档的 CSS 霞光都不敢做位移动画)。

本次没动,因为在开发机(ProMotion)上测不出掉帧 —— rAF 采样 4 秒稳定 180fps、0 janky 帧,
说明主线程很闲,瓶颈若存在也在 GPU 合成侧,需要在集显/低端机上复现才好判断收益。

可选方向:滚动期间临时把 `--glass-blur-*` 降档;或给性能设置加"减少透明度"开关(同时关流体 + 玻璃)。

### P2 CSP 只在打包路径实测过,dev 未验

`npm run dev` 下渲染层来自 `http://localhost:5173`,HMR 走 inline script + WebSocket。
CSP 里已放行 `'unsafe-inline'` 与 `ws://localhost:*`,理论上不受影响,但没实跑过。
若 dev 出现 `Refused to ...` 报错,放宽 `index.html` 的 `script-src` 即可。

### P3 榜单页首次进入会向上游发 ~38 个预览请求

并发限 4、服务端缓存 10 分钟,首次进页面约 8 秒填满。若之后发现网易限流,
把 `PREVIEW_CONCURRENCY`(`src/lib/toplist-cache.ts`)调小,或让服务端批量返回预览。

### P4 `Origin: null` / 无 Origin 仍然放行

这是必须的(渲染层自己就是这类请求),代价是本地 HTML 文件、非浏览器程序仍能调本地 API。
真正的收口要给 API 加一次性 token(主进程生成 → preload 注入 → server 校验),
改动面覆盖所有端点与 `<audio src>`/`<img src>` 直连地址,本次未做。

同理,`isSafeUpstreamUrl` 挡不住 DNS rebinding(域名解析到内网 IP),`server/lib/security.ts`
顶部注释已承认这一点,彻底修需要在连接建立后校验对端 IP。

### P5 `sandbox: false`

主窗口与三个悬浮窗都用 `sandbox: false`(preload 需要 Node API)。
`contextIsolation` 已开,preload 暴露面也收敛,风险可控;要进一步收紧得把 preload 改成
纯 IPC 转发再开 sandbox,属于较大改造。

### P6 其他小项

- `FrameLimiter` chunk 1.5 MB(three.js),已是懒加载 chunk,只影响首次进 3D 场景;
- 同一张封面在不同栏目会按不同尺寸各请求一次(如 200/512),总带宽仍远低于原图直出,暂不合并;
- `CardRail` 组件已无人引用且其 `onWheel` 里的 `preventDefault` 在 React passive 合成事件下无效
  (真要横滚劫持必须用原生非 passive 监听,参考 `QuickAccessRow`),本次直接删除了该组件。
